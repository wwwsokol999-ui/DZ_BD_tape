# Домашнее задание к занятию «Базы данных, их типы»

## Задание 1. СУБД

### 1.1 Бюджетирование проектов, финансовая аналитика, прогнозирование рисков

Для этой задачи лучше использовать **реляционную СУБД**, например **PostgreSQL**.

**Почему:**
- важна целостность данных;
- нужна четкая структура;
- много связей между сущностями;
- удобно строить отчеты, аналитику и выборки.

---

### 1.1.* Хеширование стало занимать длительное время, какое API можно использовать для ускорения работы?

Если хеширование работает долго, можно использовать **кэширование**, например через **Redis API**.

То есть смысл в том, чтобы не пересчитывать одно и то же каждый раз, а брать уже готовый результат из кэша.

---

### 1.2 Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM

Для **лендингов** лучше подойдет **документо-ориентированная СУБД**, например **MongoDB**.

Для **CRM** лучше использовать **реляционную СУБД**, например **PostgreSQL**.

**Почему:**
- лендинги требуют гибкости, потому что структура данных может меняться;
- CRM требует понятных связей между клиентами, заявками, сделками и менеджерами.

---

### 1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

Да, можно использовать **PostgreSQL** как одну СУБД.

**Как это реализовать:**
- CRM хранить в обычной реляционной структуре;
- данные с лендингов можно хранить в полях **JSONB**.

Но если делать более правильно с точки зрения архитектуры, то лучше разделить:
- **MongoDB** — для лендингов;
- **PostgreSQL** — для CRM.

---

### 1.3 Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее

Для этой задачи я бы выбрал **реляционную СУБД**, например **PostgreSQL**.

**Почему:**
- нужна простая и понятная структура;
- удобно хранить разделы, подразделы, документы, версии;
- легко поддерживать порядок в данных.

---

### 1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?

Да, можно использовать ту же **PostgreSQL**.

**Реализация:**
- сделать отдельную базу или отдельную схему;
- хранить материалы в отдельных таблицах;
- при необходимости использовать **JSONB** для дополнительной гибкости.

---

### 1.4 Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам

Для такой задачи лучше использовать **графовую СУБД**, например **Neo4j**.

**Почему:**
- здесь ключевую роль играют связи;
- нужно быстро работать с маршрутами, точками, объектами и курьерами;
- графовая БД лучше подходит для поиска путей и связей между объектами.

---

### 1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?

Да, подключить отдел закупок можно, если закупки тесно связаны с логистикой.

Но если у закупок есть своя отдельная логика:
- договоры;
- счета;
- бюджет;
- поставщики;
- акты,

то лучше сделать для них отдельную **реляционную БД** и связать ее с системой логистики.

---

### 1.5 Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

Да, теоретически можно использовать одну **PostgreSQL**.

**Почему:**
- она универсальная;
- подходит для реляционных данных;
- поддерживает **JSONB** для гибких структур;
- позволяет решать аналитические задачи.

Но если делать нормальную архитектуру, то лучше разделить:
- **PostgreSQL** — финансы, CRM, база знаний;
- **MongoDB** — лендинги;
- **Neo4j** — логистика и маршруты.

---

## Вывод

Если коротко, то **одной PostgreSQL можно закрыть почти все задачи**, но это будет компромисс.

Если делать архитектуру правильно, то лучше использовать разные типы СУБД под разные задачи:
- **PostgreSQL** — для структурированных данных и аналитики;
- **MongoDB** — для гибких данных;
- **Neo4j** — для задач, где важны связи и маршруты.
